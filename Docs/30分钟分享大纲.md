## 图像总原则

这套 PPT 建议坚持一个统一的视觉原则：

- 全场只维护一张“世界模型”：`A → f(x) → B`
- 后面所有页都不是“换图”，而是在这张图上“长出新结构”
- 每一页都回答一个问题，图只做一件事：把这一页新增的认知长出来
- `f(x)` 始终是全场主角，不要被 ShareCode、PSO、ShaderArchive 这些概念替代

### 建议的 4 种图像动作

- **展开**
  - 把一个黑盒拆开
  - 例子：把 `f(x)` 展开成 `ShaderCode + 管线状态`
- **分叉**
  - 把单一来源变成多个来源
  - 例子：从“一个 Material”变成“多个 Material”
- **汇聚**
  - 把多个东西汇总到一个全局节点
  - 例子：多个 Material 汇到 `ShaderCodeLib`
- **外包**
  - 在已有结构外面长出更高层的系统
  - 例子：在 `f(x)` 外面长出 `PSO 收集 / 构建 / 使用` 回路

### 建议的 6 个视觉层级

- `Lv0`
  - 只有 `A → f(x) → B`
- `Lv1`
  - 展开 `f(x) = ShaderCode + 管线状态`
- `Lv2`
  - 长出 `Material → ShaderCode → f(x)`，对应 Inline 直觉
- `Lv3`
  - 把单个 Material 改成多个 Material，再汇到 `ShaderCodeLib → f(x)`，对应 ShareCode
- `Lv4`
  - 把 `f(x)` 整体圈起来，告诉听众这就是运行时真正被使用的完整管线语义，也就是 PSO 的入口
- `Lv5`
  - 在 `f(x)` 外面长出 `收集 → 构建 → 使用` 回路，形成 UE 的 PSO 生命周期

## 详细大纲

### 1. 为什么客户端也要关心 PSO

**建议时长**：2 分钟

**核心信息**：
- PSO 不只是渲染组的问题，它会直接影响首帧卡顿、场景切换卡顿、资源包体、平台适配成本。
- 客户端开发里很多“看起来像内容问题”的卡顿，底层都可能和 Shader/PSO 相关。

**一句话开场**：
> 今天不是想把大家都变成渲染工程师，而是希望大家理解：PSO 为什么会影响你们每天遇到的性能、包体和稳定性问题。

---

### 2. 从第一性原理出发：A → f(x) → B

**建议时长**：3 分钟

**核心信息**：
- 渲染本质上是一个函数调用。
- A 是输入数据，f(x) 是 Shader 加状态，B 是输出结果。
- 先把“计算做什么”和“计算怎么做”分开。

**推荐表达**：
- A：顶点、法线、UV、颜色
- f(x)：Shader 程序
- State：Blend、Depth、Rasterizer、RenderTarget Format
- B：屏幕像素、深度、GBuffer、ShadowMap

**这一页的目标**：
- 让不熟悉图形 API 的听众也能跟上后面的抽象。

---

### 3. OpenGL vs Vulkan / Metal：区别到底在哪

**建议时长**：4 分钟

**核心信息**：
- OpenGL 是零散设置状态，驱动在 Draw 时隐式组装完整管线。
- Vulkan / Metal / D3D12 要求提前把 Shader 和 State 打包成完整对象，也就是 PSO。
- 所以 PSO 不是 UE 自己发明的“额外概念”，而是现代图形 API 的显式要求。

**要讲清的对比**：
- OpenGL：运行时隐式组装，可能首次 Draw 卡顿
- Vulkan / Metal：提前创建重，运行时绑定快

**一句话结论**：
> 现代 API 不是消灭了编译成本，而是把这件事前置和显式化了。

---

### 4. 映射到 UE：这套模型在引擎里长什么样

**建议时长**：4 分钟

**核心信息**：
- A 在 UE 里对应 VertexFactory
- f(x) 对应 FShader 和 Render State
- B 对应 RenderTarget / GBuffer / ShadowMap

**建议讲法**：
- 用“UE 不是魔法，只是把这些对象组织起来了”来降低理解门槛。
- Material 不等于一个 Shader 文件，它会编译出一组 ShaderMap。

**可以点到的关键词**：
- UMaterial
- FMaterialResource
- ShaderMap
- VertexFactory

---

### 5. UE 为什么要从 InlineCode 走向 ShareCode

**建议时长**：4 分钟

**核心信息**：
- InlineCode：ShaderCode 直接跟着 uasset 存
- 问题 1：重复存储，包体膨胀
- 问题 2：没有全局 Hash → Code 的查找能力
- ShareCode：把 ShaderCode 拉到全局 Archive 里，形成统一索引

**要强调的因果关系**：
- 不是“为了省一点包体所以顺便改了存储方式”
- 而是“因为后面要做全局 PSO 预编译，所以必须要有全局 ShaderCode 索引”

---

### 6. PSO 为什么必须存在

**建议时长**：4 分钟

**核心信息**：
- PSO 本质上是 Shader 和 State 的完整打包。
- 创建 PSO 是重操作，因为驱动要做编译、链接、优化、平台相关处理。
- 不提前做，这个成本就会在运行时暴露成卡顿。

**讲法建议**：
- 不要把 PSO 讲成“缓存文件”
- 先讲 PSO 是“必须做的一次计算”
- 再讲缓存只是把它前置或复用

**一句话结论**：
> PSO 不是优化技巧，而是现代图形管线里无法逃避的一步计算。

---

### 7. PSO 全流程：收集、构建、使用

**建议时长**：4 分钟

**核心信息**：
- 收集：运行时记录实际出现过的 Shader + State 组合
- 构建：离线生成可复用的 PSO 缓存
- 使用：运行时加载缓存，减少首次创建成本

**讲述顺序建议**：
1. 为什么不可能一开始就知道所有 PSO
2. 为什么需要运行时收集
3. 为什么收集之后还要构建
4. 为什么构建完还不一定完全命中

**这里可以点到的文件/概念**：
- `.ushaderbytecode`
- `.upipelinecache`
- ShaderArchive
- PipelineCache

---

### 8. PSO 中间哪些地方可以优化

**建议时长**：3 分钟

**核心信息**：
- 优化不是只盯“编译快一点”
- 更重要的是减少组合数、减少无意义变体、提高缓存命中

**建议讲成两层**：

#### 第一层：优化原则

- 减少 permutation
- 减少状态组合
- 控制材质和 mesh 组合膨胀
- 做 warmup
- 做平台分层缓存

#### 第二层：怎么用 benchmark 佐证

- 如果说“减少 permutation 有用”，就应该能看到同一组 shader case 下，PSO 数量和创建耗时同时下降。
- 如果说“某个优化主要改善的是驱动内部编译成本”，就应该能在 Mesa 路径下看到分阶段耗时下降，而不是只看到总时间变化。
- 如果说“某个优化最终能改善用户体验”，就应该能在 Native Vulkan / Native Metal 这类真实驱动路径下看到 API 总耗时下降。
- 如果说“压缩或资源组织策略更合理”，就应该同时看到 Windows 基线压缩率和移动端解压成本的变化，而不是只看包体大小。

**可以直接举的 benchmark 例子**：

- 同一组 Unreal 风格 shader，从 `S1_BasePBR` 到 `S4_PermutationStress`，观察 PSO 创建成本如何膨胀。
- 固定 shader 复杂度，只增加 static switch / permutation 数量，观察 Android `Mesa Vulkan` 和 `Native Vulkan` 的变化趋势。
- 在 `Mesa Vulkan` 下看阶段分布，区分到底是 frontend、optimization 还是 backend codegen 在放大。
- 在 `Native Metal` 下只看 API 层总耗时，验证优化是否真正落到设备可感知成本上。
- 对压缩方案，用 Windows 生成基线包，再用 Android / iOS 只测解压，判断“更高压缩率”是否值得。

**一句话结论**：
> 最好的 PSO 优化，往往不是把一次编译做快，而是根本不要生成那么多 PSO；而且这件事最好能被真实 benchmark 证明。

---

### 9. 从 Data / Compute 角度理解 PSO

**建议时长**：3 分钟

**核心信息**：
- 可以把 PSO 看成一类“不可避免的预处理计算”
- 以前驱动在背后偷偷做，现在显式放到工程系统里做
- 这类问题更接近 Data/Compute，而不是传统“写一段业务逻辑”

**建议表达**：
- 输入：Shader Hash、State、平台条件
- 计算：驱动/编译器生成平台可执行管线
- 输出：可复用的 Pipeline 对象或 Cache

**一句话结论**：
> 现代图形 API 的本质之一，就是把原来隐式的图形计算流程，变成显式的数据处理流程。

---

### 10. UE 里的真实坑

**建议时长**：2 分钟

**核心信息**：
- 理论都懂了，工程上还是会踩坑

**建议只讲 2 到 4 个典型例子**：
- mesh / VertexFactory 会影响 Metal PSO 数量
- 材质变体会把 PSO 数量迅速放大
- 不同平台同一份内容的 PSO 膨胀程度不一样
- 有些 Compute PSO 很难完全规避

**讲法建议**：
- 这里不展开源码，重点给听众建立“为什么这个系统复杂”的敬畏感

---

### 11. 结语：三句话收住

**建议时长**：1 分钟

**建议收束成三句**：
- PSO 不是 UE 的特例，而是现代图形 API 的必然结果。
- UE 的 ShareCode、ShaderArchive、PipelineCache，本质上是在管理这笔无法消失的计算成本。
- 真正有效的优化，不只是“编译更快”，而是“组合更少、命中更高、前置更充分”。
