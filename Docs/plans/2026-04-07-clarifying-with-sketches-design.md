# Clarifying With Sketches Design

## Goal

为 Codex 增加一个全局工作流 skill，在遇到结构设计、版式、动画、图示、页面改动等高误解成本任务时，默认先澄清理解，再输出低成本草图，草图通过后才允许进入代码实现。

## Why This Skill Exists

当前问题不是“实现能力不够”，而是“在理解还不稳定时就进入实现”，导致：

- 用户的空间意图没有被显式确认
- 草图骨架没有暴露出来，错误直到 Web 成品阶段才暴露
- 被批评后缺少分层复盘，导致同类错误重复出现

这个 skill 的目标，是把“先理解、先草图、再实现”变成强约束流程。

## Scope

这个 skill 适用于：

- 结构图
- 动画设计
- Remotion / Web 视觉页
- Draw.io / Mermaid / 草图类表达
- 有明显空间关系的页面改动
- 用户给出多个 numbered requirements 的任务

这个 skill 不替代具体实现 skill。它只负责在进入实现前建立共识。

## Core Contract

### 1. 不清晰就必须追问

如果模型对结构、层级、重点、页间演化、路线、语义映射任一部分理解不清晰，必须先明确列出不清晰点，再逐条确认。

禁止：

- 自行脑补后直接改代码
- 用“我大概懂了”跳过澄清
- 在没有确认前进入成品实现

### 2. 改代码前必须先给草图

任何涉及结构或视觉表达的修改，在首次代码改动前必须先给低成本草图。

草图要求：

- 速度优先，不追求最终审美
- 只要能画框、线、字即可
- 能在 Codex 客户端直接展示
- 如果需要落地文件，保存在 `<cwd>/ignore/`
- 草图的作用是暴露骨架，不是替代最终 Web 成品

### 3. 草图后必须逐条映射用户要求

如果用户使用了 `1 / 2 / 3 / 4`、`A / B / C`、项目符号等明确要求，草图之后必须做一段“要求映射总结”。

这段总结必须包括：

- 每一条要求在草图中如何体现
- 当前完成度大概多少
- 哪些地方仍然有风险
- 是否存在替代方案
- 如果存在多个合理方案且无法单方面决定，必须停下来让用户选

### 4. 被批评后必须分层复盘

如果用户明确表示草图不好，不能只继续改图，必须先总结经验。

经验分三层：

- `知识层`：是否误解了业务语义、结构语义、演化语义
- `草图层`：是否骨架错误、线条逻辑错误、对齐/密度错误
- `Web层`：是否动画、实现、排版、视觉质感与草图意图不一致

这个总结应优先出现在对话中。如果同一任务多轮反复被批评，可以额外把滚动经验记到 `<cwd>/ignore/` 下的临时笔记文件中。

### 5. 明确授权时可以跳过

如果用户明确说以下意思之一，则可以跳过“反复确认”和“先出草图”：

- 不需要确认
- 不需要草图
- 直接做
- 你自己发挥
- 先出成品

但即使跳过，也要在真正实现前用一句话说明你采用了什么假设。

## Output Modes

### Sketch Mode

目标：速度优先。

允许：

- 简化框图
- 占位文字
- 粗糙布局
- 仅表达骨架和关系

不要求：

- 最终配色
- 完整文案
- 复杂动效

### Web Mode

目标：效果优先。

要求：

- 在草图共识基础上放大实现
- 不能背离草图已经确认的骨架
- 如果实现阶段发现草图无法支撑最终效果，要先回退到草图层说明问题，而不是偷偷改骨架

## Default Sketch Review Template

草图展示后，默认输出：

| 用户要求 | 草图体现 | 完成度 | 风险 / 备注 |
| --- | --- | --- | --- |
| 1 | ... | ... | ... |
| 2 | ... | ... | ... |

然后补一段：

- 当前最可能还会被否掉的点
- 可选方案 A / B（如果存在）
- 是否需要用户拍板

## Storage Convention

如果草图或复盘需要落地文件，默认放在：

- `<cwd>/ignore/`

建议文件类型：

- `.svg`
- `.png`
- `.md`

这些文件属于过程资产，不应混入正式产物目录。

## Success Criteria

这个 skill 生效后，模型应该具备以下行为：

- 在不清晰时优先暴露疑点，而不是沉默假设
- 在进入代码前给出低成本草图
- 草图后能逐条映射用户的 numbered requirements
- 草图被否后能分层总结，而不是只做局部修补
- 用户明确授权跳过时，能直接执行，但会说明假设
