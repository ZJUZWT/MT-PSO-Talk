# 第 03 页

## 页面目标

保留第二页底部 `VertexData -> GPU -> Pixels` 主轴，但把整组结构下移，为 GPU 上方让出一条新的配置带，开始讲 GPU 不是孤立工作的，它还要接收编译结果和状态配置。

## 页面改动表

| 项目 | 内容 |
| --- | --- |
| 页面标题 | Compile / States -> GPU |
| 页面副标题 | 从单一 GPU 节点展开出上方配置层 |
| 当前页核心信息 | GPU 不只是吃 `VertexData`，还要接收 `ShaderBinary` 和状态配置 |
| 与上一页关系 | 不是改画布，而是把第二页整组往下压，在上方插入一层新的来源 |
| 这一页新增了什么 | 左侧 `ShaderCode -> ShaderBinary` 横向链路、右侧 `Depth` / `Blend`、带编号的 `Graphics API` 列表 |
| 这一页删除了什么 | 没有删除第二页主轴，只是把主轴下移并让 GPU 放大 |
| 这一页保留了什么 | 底部 `VertexData -> GPU -> Pixels` 主轴、水平主阅读骨架、GPU 仍然是全页收口点 |
| 需要强调的元素 | 上方多条线都向下收口到 GPU，且 `GFXAPI CALL` 的颜色与底部数据流颜色不同 |
| 暂不实现 / 暂不出现 | 更细的状态分类、更多输入资源、PSO 统一收口对象 |

## 动画剧本

| Beat | 时间 | 画面变化 | 屏幕元素 | 讲述词 / 备注 |
| --- | --- | --- | --- | --- |
| 1 | 02 -> 03 过渡开始 | 底部 `VertexData -> GPU -> Pixels` 整组向下移动 | 底部三段主轴 | 先保住观众对第二页的空间记忆 |
| 2 | 过渡中段 | GPU 同步放大，变成更明确的收口对象 | 中间 GPU | 让观众意识到它要接收更多东西 |

| 4 | 过渡后段 | 右边出现 `Depth`、`Blend`，两条线从上往下接入 GPU，并带上编号 | 上方右侧 | 强调这些不是运行时像素流，而是 CPU 通过图形 API 配置 GPU |
| 5 | 过渡结束 | 停在第三页终态 | 全页 | 为后续更复杂的对象打包做准备 |

## Remotion 实现备注

| 项目 | 内容 |
| --- | --- |
| 需要新增的节点 / 元素 | `ShaderCode`、`ShaderBinary`、`Depth`、`Blend`、带编号的 `Graphics API` 文本列表 |
| 需要复用的元素 | 第二页底部主轴、GPU 的中心收口位置、左右节点位置 |
| 是否需要 draw.io 草稿 | 当前可直接在网页里验证，因为结构还不复杂 |
| 是否需要镜头推进 / 缩放 | 不需要镜头推进，只在图内做节点位移和放大 |
| 是否需要逐步显隐 | 需要，先移底部主轴，再出现上方配置带 |
| 风险点 | 如果 Shader 侧和状态侧没有明确左右分区，或者编号箭头和左侧 API 名称对应不上，第三页就会讲不清楚 |
