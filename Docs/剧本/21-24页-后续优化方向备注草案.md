# 第 21-24 页后续优化方向备注草案

> 用途：先把后续几页要讲的优化方向固定下来，后续做正式动画页时直接按页拆分实现。
> 当前状态：内容备注草案（非正式动画 ledger）。

## 总体叙事主线

- 先讲“为什么要考虑卸掉 code 压缩”，把压缩/解压链路的真实成本讲透。
- 再讲缓存策略：`IO 换空间` 的基础思路（`BCache`、`LRU/LFU`、`mmap`、`circular/ring`）。
- 然后讲“编译加速”两条主线：并行化（多线程）和缩小编译集合（`UsageMask` 分桶）。
- 最后讲 `Metal` 与 `OpenGL PSO` 差异来源：核心是“后端状态来源与显式程度不同”。

## Page 21：卸掉 code 压缩（压缩/解压链路）

### 页面目标

- 说明：当前链路里“压缩 + 解压”并不是免费动作，可能吞掉启动阶段收益。
- 给出“何时应考虑卸掉压缩”的判断标准，而不是绝对结论。

### 建议画面结构

- 左：`Compressed Shader Code`。
- 中：`Decompress`（CPU 时间 + 内存带宽）。
- 右：`Compile/Load`。
- 底部放一个对比条：`压缩省 IO` vs `解压增 CPU`。

### 讲述要点

- 压缩的收益主要在包体与磁盘/下载 IO。
- 解压的成本主要在启动阶段 CPU 与内存带宽抢占。
- 当存储 IO 不是瓶颈、CPU 已是瓶颈时，卸掉压缩可能更优。

### 一句话结论

- `压缩不是默认最优，关键看瓶颈在哪一端（IO 还是 CPU）。`

## Page 22：BCache 基础策略（IO 换空间）

### 页面目标

- 给观众一个“工程上可操作”的缓存基础框架，而不是抽象概念。

### 建议画面结构

- 上半：`BCache` 组成（热数据区、冷数据区、索引）。
- 下半：三项策略并列卡片：
  - `LRU/LFU`（淘汰策略）
  - `mmap`（减少拷贝、按需映射）
  - `circular/ring`（顺序写优化）

### 讲述要点

- 本质是 `IO 换空间`：用可控内存占用换更少随机 IO。
- `LRU/LFU` 决定“谁留在热区”。
- `mmap` 适合大文件与随机访问场景。
- `circular/ring` 适合流式写入与顺序消费。

### 一句话结论

- `缓存不是越大越好，而是“命中模式 + 介质特性 + 写读形态”共同决定。`

## Page 23：编译加速（多线程 + UsageMask）

### 页面目标

- 把“编译慢”拆成可治理问题：`并行化` 和 `减少无效编译`。

### 建议画面结构

- 左：单线程基线时间线。
- 右上：多线程 worker 并行编译（队列分发）。
- 右下：`UsageMask` 分桶（按场景/材质使用位筛选，减少编译集合）。

### 讲述要点

- 多线程优化的是吞吐，但要处理任务粒度、依赖和抖动。
- `UsageMask` 优化的是“编译集合大小”，避免“能编的全编”。
- 两者组合：`更少的任务 × 更快的执行`。

### 一句话结论

- `先减集合，再做并行，通常比盲目加线程更稳。`

## Page 24：Metal 与 OpenGL PSO 差异来源

### 页面目标

- 解释差异来源，不只报现象：为什么两端编译数量/行为会不同。

### 建议画面结构

- 左列：`OpenGL PSO 路径`（更多驱动隐式状态）。
- 右列：`Metal PSO 路径`（更多显式状态组合）。
- 中间放“状态来源”桥：`Shader + Fixed/Render State` 的来源差异。

### 讲述要点

- `OpenGL` 中部分状态由驱动隐式托管，统计口径常更“收敛”。
- `Metal` 侧状态更显式，组合空间更可见，也更容易放大编译数量。
- 差异本质不是“谁更好”，而是状态表达模型不同。

### 一句话结论

- `差异来源于状态显式程度与状态来源路径，不是单一 API 性能结论。`
