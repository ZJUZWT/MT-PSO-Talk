# 线条排版评估 Benchmark

## 目的

这份文档不是讲“怎么画图”，而是讲“什么图不能交”。

针对本项目这种技术分享结构图，尤其是：

- `A -> f(x) -> B`
- OpenGL 多状态输入
- Vulkan / `PSO` 收拢
- UE `Material / ShaderCode / ShaderCodeLib / PSO`

需要一套比通用审美更硬的判断标准。

原因很简单：

- 通用 UI 审美不等于技术结构图审美
- 颜色和圆角可以掩盖一点丑，但救不了错误的走线骨架
- 技术分享图最重要的不是“丰富”，而是“可读 + 有秩序 + 有工程感”

---

## 核心结论

以后第一版图，不允许只看“节点差不多对了”。

必须先过这 3 个硬门槛：

| 门槛 | 要求 |
| --- | --- |
| 骨架正确 | 页面必须先有主骨架，再允许加细节线 |
| 走线统一 | 同一页所有线条必须遵守同一种路由语言 |
| 视觉可判读 | 观众 2 秒内必须看出主流程、次流程、强调流程 |

只要有一个不过，就不能进入“调颜色”“调动画”阶段。

---

## 空间性质优先

这套图真正先要判断的，不是“某个 node 离某条线几像素”，而是整页空间是什么性质。

元素间距、线条缝隙、对齐方式，都只是这些空间性质落地后的表象。

如果空间性质判断错了，后面就算把每个元素都调得很整齐，图还是会丑，而且会越调越僵。

以后先判断下面这些空间性质，再谈元素：

| 空间性质 | 问题 |
| --- | --- |
| 主线通道 | 本页有没有一条必须被优先保通的主线，它是不是拥有连续、干净、不被打断的通道 |
| 承接面 | `GPU / PSO` 这类核心节点是不是被当成“接收平面”来设计，而不是一个随便摆着的盒子 |
| 空间编织 | 左右两组信息是被硬切开，还是通过高低层次形成了可控穿插 |
| 密度梯度 | 哪一组应该更紧，哪一组应该更松，整页有没有疏密节奏 |
| 台阶层次 | 相关节点之间是否通过高低差形成顺序，而不是只靠左右排开 |
| 光学呼吸 | 靠近的对象之间是否仍然保留“像设计过”的小呼吸，而不是机械贴边 |

补充原则：

- 不要先问“这两个盒子之间留多少像素”，先问“这两个对象在空间上是并列、穿插、从属，还是让位关系”。
- 不要先问“线怎么绕过去”，先问“是不是应该先改承接面、改层次、改密度，让线自然变短”。
- 不要把 `node-node`、`node-line`、`line-line` 当成三张孤立检查表；它们都只是空间秩序的局部投影。

## 一票否决项

出现以下任意一项，直接判定为“不合格”，不需要继续打分：

| 编号 | 否决项 | 说明 |
| --- | --- | --- |
| V1 | 同页存在多种走线语言 | 有的先横后竖，有的先竖后横，有的贴框，有的不贴 |
| V2 | 同类输入没有共用汇线 | 比如多个 State 各自绕路去找 `GPU` / `PSO` |
| V3 | 拐点贴着框边发生 | 看起来像临时绕过去，不像设计过 |
| V4 | 强调色到处都是 | 橙线不是重点，而是泛滥 |
| V5 | 线条穿越文本核心阅读区 | 用户先看见线噪音，再看见结构 |
| V6 | 同页存在明显无意义绕路 | 明明能一折到位，却多折两次 |
| V7 | 主节点位置飘 | 相邻页中 `GPU` / `PSO` / 主函数中心不稳定 |
| V8 | 输入输出主线不清楚 | 观众看不出“从哪到哪” |

## 六维评分

每一页都按 6 个维度打分，每项 `0-5` 分。

### 1. 骨架清晰度

| 分数 | 标准 |
| --- | --- |
| 5 | 主流程一眼可见，其他线都是附属 |
| 4 | 主流程清楚，但次流程略微抢眼 |
| 3 | 看得懂，但需要扫视几次 |
| 2 | 主次不清 |
| 1 | 只能靠读文字理解 |
| 0 | 图本身无法承担说明任务 |

### 2. 走线一致性

| 分数 | 标准 |
| --- | --- |
| 5 | 所有线条遵循统一路由规则 |
| 4 | 基本统一，偶有小瑕疵 |
| 3 | 能看出规则，但执行不稳定 |
| 2 | 多套规则混用 |
| 1 | 几乎是临时拼接 |
| 0 | 没有规则可言 |

### 3. 对齐与节奏

| 分数 | 标准 |
| --- | --- |
| 5 | 节点、汇线、主轴关系非常稳定，有节奏感 |
| 4 | 整体协调，局部略松 |
| 3 | 可接受，但不精致 |
| 2 | 有明显歪斜或松散感 |
| 1 | 元素像漂浮摆放 |
| 0 | 完全失衡 |

### 4. 连接经济性

| 分数 | 标准 |
| --- | --- |
| 5 | 每条线都必要，且路径最短或接近最短 |
| 4 | 大多简洁，个别稍绕 |
| 3 | 能工作，但冗余开始明显 |
| 2 | 有多条“为了连上而连上”的线 |
| 1 | 线条数量和拐点明显过多 |
| 0 | 连线本身成了噪音主体 |

### 5. 强调控制

| 分数 | 标准 |
| --- | --- |
| 5 | 当前页唯一重点被精准强调，其他都退后 |
| 4 | 重点清楚，但强调略多 |
| 3 | 重点存在，但竞争者太多 |
| 2 | 强调逻辑模糊 |
| 1 | 所有东西都在喊“看我” |
| 0 | 根本没有重点 |

### 6. 页间稳定性

| 分数 | 标准 |
| --- | --- |
| 5 | 相邻页像同一骨架演化出来 |
| 4 | 大体稳定，局部换位较多 |
| 3 | 能对应，但有“换图感” |
| 2 | 需要重新适应布局 |
| 1 | 上一页和下一页像两个系统 |
| 0 | 完全无法连续观看 |

---

## 最低通过线

| 条件 | 要求 |
| --- | --- |
| 单项最低分 | 任何一项不得低于 `3` |
| 总分 | 不低于 `24 / 30` |
| 一票否决项 | 不得触发 |

如果没有达到：

- 不允许进入调色阶段
- 不允许进入动画阶段
- 必须先重排骨架和汇线

## 走线硬规则

这是本项目之后所有结构图默认遵守的规则。

| 规则编号 | 规则 |
| --- | --- |
| R1 | 默认只用横平竖直的正交走线 |
| R2 | 同类输入优先接入共用总线，再进入核心节点 |
| R3 | 同一类节点从同一侧进入核心节点 |
| R4 | 拐点不得贴框，必须保留固定安全距离 |
| R5 | 能一折解决，不许两折；能两折解决，不许三折 |
| R6 | 强调色只给“本页主动作”那一段 |
| R7 | 主节点优先对齐在稳定中轴上 |
| R8 | 页面扩展优先平移节点，不要先让线扭曲 |
| R9 | 线条尽量不跨越标题、说明、重点阅读区 |
| R10 | 多页演化时，骨架优先稳定，细节其次 |
| R11 | 跨页继承的主骨架默认保持中性，除非它正是本页要强调的新动作 |
| R12 | 如果一页存在唯一明确主线，优先把它做到最少弯折；能零弯折就不要一弯折 |
| R13 | 当两组节点横向争空间时，优先拉开纵向层次，让它们发生可控穿插，而不是先把整组硬推远 |
| R14 | “贴近”是光学关系，不是零缝隙接触；相关盒子与相邻竖线 / 盒边之间应保留稳定小缝隙 |
| R15 | 盒子与盒子之间同样必须保留稳定小缝隙；不允许节点彼此相切、压边或靠到像误碰 |

## 本项目特化规则

### OpenGL 页

| 项 | 要求 |
| --- | --- |
| 主节点 | `GPU` 固定在右侧主轴 |
| 上游主链 | `ShaderCode -> Compile -> Shader Binary` 必须是一条清晰水平链 |
| 主线优先级 | `Shader Binary -> GPU` 如果是本页真正主提交流程，优先做成零弯折；其他输入为它让路 |
| 继承骨架 | `Vertex Data -> GPU` 属于从 `03` 继承下来的稳定骨架，默认可保持灰线，不和本页新增动作争主色 |
| 状态输入 | `Blend / Depth / Raster` 要保持“分别设置、分别喂入”的语义；可以统一方向与节奏，但不能先被画成语义上的共享总线 |
| 强调色 | 只强调最终 `GfxAPI` 提交段，不强调所有线 |
| 禁止项 | 三条状态线各自不同高度、不同拐法直接冲进 `GPU`；也不能为了排版方便先被画成“统一收拢后再进入 `GPU`” |

### Vulkan 页

| 项 | 要求 |
| --- | --- |
| 主节点 | `PSO` 是中心，`GPU` 是下游结果 |
| 输入组织 | `Shader Binary` 与 `State` 必须表现为“收拢” |
| 汇聚方式 | 多输入优先先收束到 `PSO` 外围统一入口 |
| 强调色 | 只强调 `PSO -> GPU` 或 `vkCreateGraphicsPipelines` 对应主动作 |
| 禁止项 | 每个输入各走各线，各找各的入口 |

### 跨页对比规则

| 项 | 要求 |
| --- | --- |
| 演化优先级 | 多页连续演化时，先保留上一页主骨架，再引入这一页的新结构变化 |
| 对比边界 | 如果下一页的核心变化是“收拢 / 汇聚 / 结构化对象”，上一页不能为了排版方便提前画出同类收拢结构 |
| `04 -> 05` 特例 | `04 OpenGL` 必须表现“零散提交、分别设置”；`05 Vulkan / PSO` 才第一次表现“先收拢成 `PSO` 再进入 `GPU`” |
| 禁止项 | 不能让 `04` 里的总线语义看起来已经像 `05` 的 `PSO` 前身，否则页间对比会塌掉 |

### UE 页

| 项 | 要求 |
| --- | --- |
| 叙事核心 | 是映射和收敛，不是重新发明一套新图 |
| `Material` 页 | `Material -> ShaderCode` 必须是最强主线 |
| `ShaderCodeLib` 页 | 多个 `Material` 必须先共享，再进入主干 |
| `UE PSO` 页 | 视觉上要和 Vulkan 页呼应，形成闭环 |
| 禁止项 | 每页节点位置大换血，导致观众失去空间记忆 |

## 常见丑图模式

以后只要出现这些模式，就要主动报警。

| 模式 | 特征 | 应对 |
| --- | --- | --- |
| 临时绕路型 | 某条线为了绕开东西突然拐很多次 | 先移动节点，再重画线 |
| 各找各妈型 | 多个输入都各自冲向核心节点 | 引入共享总线 |
| 贴边碰瓷型 | 线贴着框边转折或擦框而过 | 统一安全距离 |
| 强调滥用型 | 橙线到处都是 | 只保留一个主强调动作 |
| 骨架漂移型 | 相邻页主节点位置变化大 | 锁定主轴 |
| 信息挤爆型 | 想表达太多，导致线变脏 | 拆页或降维表达 |

## 自检流程

每次做一页结构图，必须按这个顺序检查：

1. 先看不看文字，能不能一眼认出主流程。
2. 再只看线，不看框，判断走线规则是否统一。
3. 再把所有强调色临时想象成灰色，看骨架是否仍然成立。
4. 再和上一页并排，判断主节点有没有无意义漂移。
5. 最后才看配色、圆角、动画。

如果第 1 到第 4 步过不了，第 5 步一律不做。

## 出图前强制问答

在真正开始拖节点之前，先把下面 6 个问题写出来；答不出来就不要开始画。

1. 上一页要保留哪条骨架，本页真正新增的结构变化又是什么？
2. 这一页哪一条是主线？它的目标弯折数是多少？有没有被配角逼弯？
3. 哪些线是“本页主动作”，哪些线只是“继承骨架”或“结构收口”？
4. 哪个节点是承接面？它的宽度 / 高度够不够让上游各自拥有落点？
5. 两组元素如果在抢横向空间，能不能通过拉大纵向层次，让它们横向穿插约三到四成面积？
6. 哪些对象应该靠让位解决，哪些对象不允许挤压主线？
7. 如果把所有橙线临时改成灰线，骨架还成立吗；如果只保留主线，故事还说得清吗？

## 出图后快速校验

每画完一版，至少做下面 5 个硬检查：

1. 数主线弯折数：主线如果还能再少一个弯，就说明布局还没收干净。
2. 数强调色条数：每一条橙线都要能回答“为什么它比灰线更值得看”。
3. 看承接面：`GPU / PSO` 之类的接收节点是不是已经宽到能承住上游，不是在窄边上硬挤。
4. 看空间编织：相关节点组之间有没有足够纵向层次，从而产生紧凑但不打架的横向穿插。
5. 看光学缝隙：所有“贴近”关系是否仍保留小缝，不存在零距离相切、压线、擦边或文字碰撞。
6. 看节点缝隙：相邻 node 之间是否也保留了稳定小缝，而不是只顾着 node 和线的距离。
7. 看页间对比：这一页有没有提前画出下一页才该出现的结构语义。
8. 看预览图而不是 XML：必须导出位图快速扫一眼，确认空间感和线条节奏真的成立。

## 给模型的硬提示词

以后在做这类图之前，应该先给自己加这一段约束：

> 这是技术结构图，不是自由构图海报。  
> 先设计骨架，再设计节点，再设计走线，再设计强调色。  
> 所有线必须遵守统一正交路由；同类输入必须共用汇线；拐点不能贴框；强调色只能有一个主动作。  
> 如果图看起来像“线为了连上而连上”，直接判失败并重排。

## 当前问题复盘

这次版本的问题，不是“审美还差一点”，而是前置流程漏了这一关：

| 失误 | 本质 |
| --- | --- |
| 太早进入实现 | 还没把骨架路由标准写死 |
| 把“内容正确”误当成“图可交付” | 技术图不是信息对就行 |
| 过度相信通用设计感觉 | 结构图需要专门的路由纪律 |

所以以后应该这样改：

| 旧流程 | 新流程 |
| --- | --- |
| 想内容 -> 直接画 | 想内容 -> 先过线条 benchmark -> 再画 |
| 节点先放上去再补线 | 先定总线和主轴，再挂节点 |
| 线丑了再修 | 线不合格直接返工骨架 |

## 结论

你想让我“第一版就意识到”，靠的不是更多形容词，而是更硬的规则。

以后这类页面，如果我没有先通过这份 benchmark，就不应该把图交出来。
